# --trailer support for commit subcommand

## Problem

safegit's `commit` subcommand doesn't support `--trailer "Key: Value"`. Git itself supports this natively since git 2.32+. Tools like rlsbl need to add trailers to commits for structured metadata (e.g., `Autogenerated: true` to mark auto-generated commits).

## Expected behavior

```
safegit commit --trailer "Autogenerated: true" -m "message" -- file1
```

This should produce a commit with the trailer appended in the standard git trailer format (after a blank line separator in the commit message).

Multiple trailers should be supported:

```
safegit commit --trailer "Autogenerated: true" --trailer "Tool: rlsbl" -m "message" -- file1
```

## Current workaround

Embed the trailer directly in the message body. Git recognizes `Key: value` lines after a blank line as trailers, so passing a multi-line message works:

```
safegit commit -m "message

Autogenerated: true" -- file1
```

This is fragile and inconsistent with how safegit already supports `-m` and `-F` as proper flags.

## Implementation

Parse `--trailer` flag (repeatable), append each trailer to the commit message before invoking git. Alternatively, pass `--trailer` through directly to the underlying `git commit` call, since git handles trailer formatting natively.

## Effort

Small. Parse the flag, pass it through to git (or append to message before commit). No architectural changes needed.
